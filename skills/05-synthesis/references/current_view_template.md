## 一、领域定义与问题框架

本章是整个手册的基石——清晰定义"这是什么领域、要解决什么问题、为什么重要"。

### 1.1 核心研究问题

用 1-2 段话完整描述本领域要解决的核心问题：
- 在什么**应用场景**下（具体系统/业务场景）？
- 存在什么**核心矛盾**（工程约束、物理限制、设计权衡）？
- 要达成什么**目标**（量化指标，如果有文献基线则注明）？

### 1.2 为什么这个问题重要

- **工程意义**：这个问题对实际系统/业务有什么影响？（量化——有多少系统受影响、影响程度多大）
- **学术价值**：这个问题的解决会推动什么方向的认识进步？与哪些更大的研究趋势相关？

### 1.3 技术背景

简要说明本领域涉及的核心技术概念，使非该细分方向的系统研究者也能理解后续讨论。逐项解释关键术语、缩写、基础原理。

### 1.4 边界定义

明确本手册覆盖的范围和不覆盖的范围（如：覆盖 LLM 推理扩容的存储面加速，不覆盖训练场景的网络优化）。

## 三、现有方法的不足与 gap 分析

**这是申请书"立项依据"部分的核心素材来源。** 每个 gap 必须具体到可以直接写进申请书的程度。

### 3.1 已确认的核心 gap（证据强度：强/中-强）

逐项深度分析。每条格式：

**Gap X：[简短标题]**

- **具体描述**：在 [具体场景] 下，当 [具体约束条件] 时，现有方法 [具体失效模式]，导致 [具体后果]。不能写成"还有改进空间"这样的空话。
- **证据链**：（文献 A → 发现 X → 文献 B → 确认 Y → ...）——证明这个 gap 确实存在且重要的独立证据来源。每条证据标注来自哪篇论文的什么实验/结论。
- **为什么现有方法解决不了**：（不是因为"没人做过"，而是因为现有方法的设计假设与此场景的约束条件根本冲突）——具体分析最近邻的工作在这个场景下为什么不行。
- **对本课题的意义**：这个 gap 是否适合作为本课题的核心切入点？适合/不适合的理由。

### 3.2 待验证的 gap（证据强度：中/弱）

格式同上，但需明确标注哪些环节缺乏证据、需要什么方向的调研来确认。

### 3.3 已排除的 gap

- [描述] — 排除原因：[具体论文/数据的推翻依据]
